## 反臆造守则

你输出的每一句话必须基于输入数据中真实存在的信息。
具体规则：

1. 只引用输入数据中明确出现的事实、数字、案例、原话
2. 不要根据发言者身份脑补“他可能在说什么”
3. 不确定时宁可不写，也不要写
4. 不要用“他指出”“她强调”等包装不确定细节

【自检】
输出前问自己：这条 signal 里的每个具体细节，能不能在输入原文里找到？
找不到的删掉。

你要先做“信号抽取”，不是直接生成主题。

对于每条 builder X 帖子，输出 1 条 signal，字段如下：
- `content_id`
- `source`：发言者名字
- `url`
- `topic_label`：8-16 字中文标签，只写这条帖子最核心的议题
- `core_claim`：用 1 句话概括这条帖子到底在说什么，20-40 字中文
- `angle`：从以下枚举里选 1 个：`技术机制` / `具体案例` / `商业影响` / `个人体验` / `未来判断`
- `excerpt`：1 句中文事实陈述，20-40 字，保留帖子里最能支撑主题判断的具体信息
- `spotlight_text`：给“今日热议”降级展示用的中文句子，25-45 字，要像正常人在转述“这条帖子讲了什么”

规则：
- 每条帖子只允许 1 个 `topic_label`
- 不要把“共享关键词”当成同一议题
- `topic_label` 要尽量具体，例如“代码审查闭环”优于“AI 代理”
- `excerpt` 只写最核心的新信息，不写评价句
- `spotlight_text` 必须通顺，允许比 `core_claim` 更口语一点
- `spotlight_text` 必须有主谓结构，不要写成参数说明或标签拼贴
- 反例：`每次提交后自动审查并修复代码错误，最多循环5次`
- 正例：`Peter Steinberger 现在让 Codex 在每次提交后自动审查代码，发现问题就继续修，最多循环 5 次`
- 如果原帖信息太弱，只保留最稳妥的核心主张

输出 JSON 结构：
{{
  "signals": [
    {{
      "content_id": "...",
      "source": "...",
      "url": "...",
      "topic_label": "...",
      "core_claim": "...",
      "angle": "...",
      "excerpt": "...",
      "spotlight_text": "..."
    }}
  ]
}}

严格输出 JSON，不要用 markdown 代码块包裹。

【输入数据】
Builder X 帖子（共 {n_posts} 条）：
{builder_posts}
